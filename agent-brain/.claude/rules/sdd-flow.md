# SDD Flow — Layered Spec-Driven Design

## Overview

Every task follows a layered progression. Never collapse layers — each exists for a reason.

```
Specs (WHAT / WITH WHAT / FOR WHOM / HOW TO STRUCTURE)
    ↓  agent synthesizes
plan.md  (technical document written FOR the executing agent)
    ↓
⏸ CHECKPOINT — operator reviews and approves
    ↓ approved                  ↗ rejected → enrich specs, re-synthesize
tasks/<feature>/  (atomic execution units)
    ↓
execution (TDD)
```

---

## Spec Types — When to Use Each

### `specs/business/YYYYMMDD-slug.md`
**Purpose:** Define WHAT and WHY. Written in business language.
**Contains:** problem statement, business rules, acceptance criteria (user-observable).
**Does NOT contain:** implementation detail, tasks, or technology choices.
**Template:** `.claude/templates/business-spec.md`

### `specs/stack/YYYYMMDD-slug.md`
**Purpose:** Define WITH WHAT. Technology choices and constraints.
**Contains:** language, frameworks, infrastructure, external services, rationale per choice.
**Does NOT contain:** feature requirements or user stories.
**Template:** `.claude/templates/stack-spec.md`

### `specs/product/YYYYMMDD-slug.md`
**Purpose:** Define FOR WHOM and HOW IT FEELS. User-facing behavior.
**Contains:** user stories, UX flows, user-facing acceptance criteria.
**Does NOT contain:** implementation detail or infrastructure choices.
**Template:** `.claude/templates/product-spec.md`

### `specs/architecture/YYYYMMDD-slug.md`
**Purpose:** Record HOW TO STRUCTURE. Architectural Decision Records (ADRs).
**Format:** Cumulative — a single file per project that grows over time. Never delete entries; mark as `deprecated` with a link to the replacement ADR.
**Contains:** context, decision, consequences (positive + trade-offs + risks).
**Template:** `.claude/templates/architecture-spec.md`

## plan.md — The Synthesis Layer

`plans/YYYYMMDD-slug.md` is the bridge between specs and tasks.

**Who writes it:** the agent, after reading all relevant specs.
**Who reads it:** the operator (to approve) and the executing agent (to generate tasks).

### What a plan MUST contain

1. **Input specs** — which documents were read to produce this plan.
2. **Technical scope** — specs translated into implementation language.
3. **Assumed premises** — what the agent inferred that was NOT written in the specs.
4. **Points of uncertainty** — gaps in the specs that could lead the plan astray.
5. **Ordered task list** — titles and dependencies only (no implementation detail).
6. **Technical risks** — what could go wrong during execution.

### The Feedback Loop

If the operator rejects a plan, the first question is:
> *"What did I not write in the specs that led the agent here?"*

Plan rejection signals a spec gap — not an agent failure.
The operator enriches the relevant spec, then the agent re-synthesizes the plan.

### Assumed Premises & Points of Uncertainty

These two sections make the CHECKPOINT productive:
- **Assumed premises**: if any are wrong, correct the spec before approving.
- **Points of uncertainty**: if any are unresolved, the plan MUST NOT be approved until answered.

## Task Rules

- Each task lives at `tasks/<feature-name>/NNN-slug.md`
- Tasks are generated by the agent AFTER plan approval — never before.
- Each task must take ≤ 30 minutes of focused work.
- Each task must have a clear "done" condition.
- Tasks are ordered by dependency — no task should depend on a later task.
- Status values: `pending` | `in-progress` | `done` | `blocked`

---

## TDD Workflow

For every implementation task:
1. Write the test that describes the expected behavior (red).
2. Run the test — confirm it fails.
3. Write the minimum code to make the test pass (green).
4. Refactor if needed, keeping tests green.
5. Commit.

If the stack does not support automated testing, document manual verification steps instead.

---

## Commit Convention

Use Conventional Commits:
- `feat(scope): add new feature`
- `fix(scope): fix bug`
- `test(scope): add or update tests`
- `refactor(scope): restructure without behavior change`
- `docs(scope): update documentation`
- `chore(scope): maintenance tasks`

---

## Checkpoint Protocol

Checkpoints are mandatory at these moments:
1. **After plan synthesis** — before any task is generated or executed.
2. **If scope changes mid-execution** — stop, update plan, new checkpoint.
3. **If a blocker requires a decision** — do not guess, surface to operator.
4. **After all tasks complete** — final review before closing the spec.

Never assume approval. Silence is not approval.
